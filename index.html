Features

A stable KSeF core for many products.

KSeF_2 owns KSeF concerns and workflow contracts. ERP details remain in independently versioned connectors.

01 / COMMUNICATE

KSeF API

Authentication, session and token lifecycle, invoice exchange, status retrieval, bounded transport retries and safe diagnostics behind one Gateway boundary.

02 / ORCHESTRATE

Workflow Engine

Company-scoped business workflows coordinate authentication, receipt, submission, refresh and synchronization without exposing protocol details.

03 / INTEGRATE

Port Contracts

Narrow, vendor-neutral contracts let a connector supply documents, receive native KSeF documents and update business-system status.

04 / EXPOSE

Public API

Transport-independent requests and typed workflow results provide a stable boundary for client applications and service hosts.

05 / PARSE

Library API

A focused parse_fa3 API exposes validated FA(3) document data without workflow, transport, profile or ERP dependencies.

06 / PROTECT

Isolation and safety

Exactly one Company Context per execution, native XML preservation, redacted outcomes and explicit ownership boundaries reduce integration risk.

Architecture

Dependencies point toward stable contracts.

Business workflows depend on Core-owned ports. A composition root binds those ports to external implementations without coupling the Core to an ERP.

INBOUND / BUSINESS SYSTEMBusiness System Provider

Implemented by an external connector. Owns all ERP-specific communication and mapping.

OUTBOUND / KSEFKSeF Gateway

The exclusive implementation boundary for KSeF protocol, authentication, sessions and API versions.

KSeF_2 has no dependency on an ERP vendor or ERP database.
Connectors implement Port Contracts; they do not change the Core.
Native KSeF XML remains the source document.
Business data stays with its external owner.
Getting started

Choose the boundary you need.

Use the Library API for FA(3) parsing or integrate full business capabilities through workflows and Port Contracts.

1

Install the project

KSeF_2 currently targets Python 3.12.

python3.12 -m venv .venv
source .venv/bin/activate
python -m pip install -e '.[dev]'
2

Parse an FA(3) document

The Library API stays independent from workflows and external systems.

from ksef_bridge.library import parse_fa3

document = parse_fa3(xml_text)
print(document.document_number)
3

Build a connector

Implement only the Business System Ports your workflow needs, then bind the implementation in the composition root.

ERP → Business System Provider
    → KSeF_2 Port Contracts
    → Business Workflows
KSeF credentials, certificates and company mappings are runtime configuration. Never place secrets or invoice XML in source control or public logs.

FAQ

Clear responsibility boundaries.

Is KSeF_2 an Optima integration?

No. KSeF_2 is ERP-independent. The Optima integration lives in the separate ksef-optima-connector project and consumes KSeF_2 as an independent dependency.

Does KSeF_2 connect to an ERP database?

No. Database access and vendor-specific APIs belong to an external Business System Provider. KSeF_2 sees only its own Port Contracts.

What is the difference between Public API and Library API?

Public API exposes company-scoped business workflows and typed results. Library API is the smaller parsing boundary for applications that only need to inspect an FA(3) document.

Can I build a connector for another ERP?

Yes. Implement the required Business System Ports, keep vendor mapping inside the connector and bind it through the composition root. The KSeF_2 Core does not need to change.

Is KSeF_2 an invoicing or accounting application?

No. It provides KSeF communication and workflow capabilities. Client applications and ERP systems retain their own business models and data ownership.
