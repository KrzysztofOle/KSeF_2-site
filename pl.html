Co robi projekt

Trzy granice. Jeden kontrolowany przepływ.

KSeF_2 oddziela publiczne możliwości, szczegóły protokołu KSeF i zachowanie ERP, dzięki czemu rdzeń pozostaje testowalny i neutralny technologicznie.

01 / KOMUNIKACJA

KSeF API

Uwierzytelnienie certyfikatem, typowane wyniki, ograniczone ponowienia i krótkotrwałe sesje za dedykowaną bramą.

02 / ZACHOWANIE

Natywny dokument

Oryginalny XML pozostaje dokumentem źródłowym. Izolacja firm, korelacja i integralność są jawne.

03 / ADAPTACJA

Comarch Optima

Struktury SQL i projekcje Optimy pozostają w Providerze opartym na kontrolowanych snapshotach, śladach i odkryciach.

Aktualny status

Uczciwe bramki, widoczne ograniczenia.

„Zaimplementowane” oznacza potwierdzone w zadeklarowanym zakresie. Wyniki badań nie stają się automatycznie zgodą na bezpośredni zapis SQL.

Workflow atomowe · UC-001–UC-008

Zaimplementowane

Warstwa atomowych workflow jest kompletna dla uwierzytelniania, odbioru, wysyłki, odświeżania statusu oraz walidacji konfiguracji i połączenia.

  • kontrakty i autoryzacja powiązane z Company
  • typowane, zredagowane wyniki
  • rzeczywiste wycinki Gateway KSeF DEMO

UC-005 · Regresja DEMO

DEMO gotowe

Runner aktualnego kodu wykonuje stabilną specyfikację SC001–SC013 przez izolowane rooty UC-005 dla KO_PROJEKT_demo i SYLWIA_demo.

  • unikalne FA(3) i bezpieczne raporty plikowe
  • SC010 i SC011 zachowane jako specification-only
  • wynik międzyfirmowy jest zbiorczy, bez dokładnej korelacji XML

Badania Optimy

Dowody

Natywne zachowanie DKF jest odtwarzane z kontrolowanych snapshotów SQL i śladów Extended Events.

  • model rekordu z 63 kolumnami
  • payloady JSON, XML i HTML
  • trwały rejestr odkryć

Bezpośredni zapis SQL

Brak zgody

Mapowanie dry-run jest użyteczne, ale współbieżność, rollback, duplikaty i akceptacja w UI nadal wymagają dowodów.

  • brak produkcyjnej ścieżki zapisu
  • przyszły zabezpieczony executor
  • akceptacja A/B względem natywnej Optimy
