Funkcje

Stabilny rdzeń KSeF dla wielu produktów.

KSeF_2 odpowiada za KSeF i kontrakty workflow. Szczegóły ERP pozostają w niezależnie wersjonowanych connectorach.

01 / KOMUNIKACJA

API KSeF

Uwierzytelnianie, cykl życia sesji i tokenów, wymiana faktur, odczyt statusów, ograniczone ponowienia i bezpieczna diagnostyka za jedną granicą Gateway.

02 / ORKIESTRACJA

Workflow Engine

Workflowy biznesowe w kontekście firmy koordynują uwierzytelnianie, odbiór, wysyłkę, odświeżanie statusu i synchronizację bez ujawniania protokołu.

03 / INTEGRACJA

Port Contracts

Wąskie, neutralne kontrakty pozwalają connectorowi dostarczać dokumenty, odbierać natywne dokumenty KSeF i aktualizować status w systemie biznesowym.

04 / UDOSTĘPNIANIE

Publiczne API

Żądania niezależne od transportu i typowane wyniki workflow tworzą stabilną granicę dla aplikacji klienckich i hostów usług.

05 / PARSOWANIE

Library API

Wąskie API parse_fa3 udostępnia zwalidowane dane FA(3) bez zależności od workflow, transportu, profili i ERP.

06 / OCHRONA

Izolacja i bezpieczeństwo

Dokładnie jeden kontekst firmy na wykonanie, zachowanie natywnego XML, zredagowane wyniki i jawne granice własności ograniczają ryzyko integracji.

Architektura

Zależności wskazują stabilne kontrakty.

Workflowy biznesowe zależą od portów należących do Core. Composition Root wiąże je z implementacjami zewnętrznymi bez uzależniania rdzenia od ERP.

Pierwsze kroki

Wybierz potrzebną granicę.

Użyj Library API do parsowania FA(3) albo integruj pełne możliwości biznesowe przez workflowy i Port Contracts.

1

Zainstaluj projekt

KSeF_2 obecnie wymaga Pythona 3.12.

python3.12 -m venv .venv
source .venv/bin/activate
python -m pip install -e '.[dev]'
2

Przetwórz dokument FA(3)

Library API pozostaje niezależne od workflow i systemów zewnętrznych.

from ksef_bridge.library import parse_fa3

document = parse_fa3(xml_text)
print(document.document_number)
3

Zbuduj connector

Zaimplementuj tylko porty systemu biznesowego wymagane przez workflow, a następnie powiąż implementację w Composition Root.

ERP → Business System Provider
    → KSeF_2 Port Contracts
    → Business Workflows
Poświadczenia KSeF, certyfikaty i mapowania firm są konfiguracją uruchomieniową. Nigdy nie umieszczaj sekretów ani XML faktur w repozytorium lub publicznych logach.

FAQ

Jasne granice odpowiedzialności.

Czy KSeF_2 jest integracją z Optimą?

Nie. KSeF_2 jest niezależny od ERP. Integracja z Optimą znajduje się w osobnym projekcie ksef-optima-connector i korzysta z KSeF_2 jako niezależnej zależności.

Czy KSeF_2 łączy się z bazą ERP?

Nie. Dostęp do bazy i API konkretnego producenta należy do zewnętrznego Business System Provider. KSeF_2 widzi wyłącznie własne Port Contracts.

Czym różni się Public API od Library API?

Public API udostępnia workflowy biznesowe w kontekście firmy i typowane wyniki. Library API jest mniejszą granicą do parsowania dokumentów FA(3).

Czy mogę utworzyć connector dla innego ERP?

Tak. Zaimplementuj wymagane Business System Ports, pozostaw mapowanie producenta wewnątrz connectora i powiąż go przez Composition Root. Core KSeF_2 nie wymaga zmiany.

Czy KSeF_2 jest programem do fakturowania lub księgowości?

Nie. Udostępnia komunikację z KSeF i możliwości workflow. Aplikacje klienckie oraz systemy ERP zachowują własne modele biznesowe i własność danych.
